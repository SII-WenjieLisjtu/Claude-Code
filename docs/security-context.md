# 公开快照与供应链安全背景

这篇文档不讨论如何利用问题，只讨论这份快照为什么值得从安全角度被认真看待。

## 一、这件事本质上是什么？

这不是传统意义上的源码仓库公开，而是一次与**构建产物、source map 与发布流程**相关的暴露事件。

它提醒我们：

- 发布出去的不只是二进制或打包文件
- source map 可能暴露源码路径、模块结构甚至完整源码位置
- 对象存储和构建中间产物如果权限不当，风险会被放大

## 二、为什么它值得研究？

因为它展示的是非常现实的一类问题：

1. **构建制品治理**不只是前端问题
2. **CLI 工具链**同样存在 source map 和制品暴露风险
3. **供应链安全**不只看依赖，也要看发布流程本身

## 三、从这个案例里能学到什么？

### 对工程团队
- 检查发布流程是否会携带不该公开的 map 文件或内部路径
- 检查对象存储中的构建中间产物是否可被外部访问
- 检查 CI/CD 是否对最终包做过安全审计

### 对安全研究者
- 这类案例很适合研究“构建元数据如何暴露系统内部结构”
- 它也说明现代 AI 工具链已经是供应链安全的重要目标面

### 对读源码的人
- 与其停留在“看到了源码”，不如进一步思考：
  - 为什么这些文件会暴露？
  - 哪个环节本来应该阻止这件事？
  - 如果你是维护者，会如何避免类似问题再次发生？

## 四、一个更有价值的角度

这份快照最有意义的地方，不是八卦，而是复盘：

> 一个复杂 AI 工程系统，在发布链路上哪里可能把内部实现意外带到外部世界。

这比单纯围观事件本身更有长期价值。
